Microsoft Azure Interview Prep (1-20)

Core architectural concepts, resource management, IAM, and cloud governance.

Azure organizes resources into four levels: Management Groups, Subscriptions, Resource Groups, and Resources. Any Azure Policy or Role-Based Access Control (RBAC) applied at a higher level is inherited automatically by all child elements below it.

Availability Sets protect applications against local hardware failures within a single datacenter by distributing VMs across Fault and Update domains. Availability Zones protect against an entire datacenter failure by placing VMs across physically separate datacenters within an Azure region.

LRS (Locally Redundant Storage) replicates data 3 times within a single datacenter. ZRS (Zone-Redundant Storage) replicates synchronously across 3 distinct zones in the region. GRS (Geo-Redundant Storage) copies data to a secondary paired region hundreds of miles away for disaster recovery.

VNet Peering seamlessly connects two Azure virtual networks over the Microsoft backbone network, allowing traffic to route privately without using public internet or VPN gateways. VNets cannot have overlapping IP address spaces to be peered.

A Network Security Group (NSG) contains stateful firewall rules applied to subnets or individual network interfaces. An Application Security Group (ASG) allows you to group VMs under a logical tag name and use that tag as a source/destination filter inside an NSG rule, avoiding explicit IP maintenance.

Conditional Access is a policy engine that evaluates contextual signals (like user identity, location, device health, and application target) to enforce security requirements such as MFA, blocks, or session limitations before granting access.

Azure Load Balancer operates at Layer 4 (Transport layer - TCP/UDP) and routes traffic solely based on IP and port. Azure Application Gateway is a Layer 7 (Application layer) load balancer that supports URL path-based routing, SSL termination, and features an integrated Web Application Firewall (WAF).

Azure Front Door is a global, scalable entry-point that uses the Microsoft global edge network to deliver high-performance, fast, and secure web applications with global Layer 7 load balancing and CDN capabilities.

A Private Endpoint is a specific private IP address allocated from your VNet to access an Azure PaaS service (e.g., Azure SQL). Azure Private Link ensures all traffic between your VNet and that PaaS service stays entirely within the private Microsoft backbone network, bypassing the public internet.

Managed Identities provide an automatically managed identity in Microsoft Entra ID for cloud applications to use when connecting to resources. This eliminates the risk of hardcoding service principal credentials or connection strings in your source code.

Azure Advisor is a personalized cloud consultant tool that analyzes your deployed resource configurations and telemetry to provide best practice recommendations across high availability, security, performance, operational excellence, and cost optimization.

RBAC regulates *who* has access permissions to manage specific Azure resources (focusing on user actions). Azure Policy regulates resource properties during deployment, evaluating *what* configurations are allowed (e.g., preventing the creation of unencrypted disks regardless of user access level).

Azure Key Vault is a centralized cloud service for securely storing cryptographic keys, application secrets, certificates, and passwords. Applications authenticate to it securely using Azure Managed Identities, preventing any secret sprawl.

Azure App Service is a fully managed HTTP-based platform-as-a-service (PaaS) meant for hosting monolithic or multi-tier web applications. Azure Functions is a serverless, event-driven compute platform engineered to run short, stateless execution snippets triggered by external hooks.

ExpressRoute lets you extend your on-premises networks into the Microsoft cloud over a dedicated, private connection facilitated by a connectivity provider. This traffic does not go over the public internet, ensuring lower latency and higher reliability.

Diagnostic logs are enabled via Diagnostic Settings on individual resources. They can be streamed directly to a Log Analytics Workspace for querying, stored securely inside an Azure Storage Account, or streamed to an Event Hub for external SIEM consumption.

Log Analytics is the primary workspace engine inside Azure Monitor that consolidates log telemetry. Kusto Query Language (KQL) is the expressive, high-performance read-only query system used to filter, aggregate, and visualize data inside those logs.

Azure Autoscale is a built-in feature that automatically scales compute resources (like Virtual Machine Scale Sets or App Services) horizontally up or down based on predefined performance metrics (such as CPU utilization or Queue depth) or calendar schedules.

Azure Blueprint enables cloud architects to define a repeatable set of Azure resources, artifacts, and guardrails—including RBAC assignments, policies, and ARM/Bicep templates—to rapidly provision compliant, standardized subscriptions across an organization.

It is a set of guiding design principles used to improve the quality of cloud workloads. It is anchored by five distinct pillars: Reliability, Security, Cost Optimization, Operational Excellence, and Performance Efficiency.

Azure DevOps Interview Prep (21-40)

CI/CD automation, YAML pipelines, agent architecture, and release strategies.

Azure DevOps comprises five primary modules: Azure Boards (agile project management), Azure Repos (Git source control), Azure Pipelines (CI/CD engine), Azure Test Plans (manual and exploratory testing), and Azure Artifacts (package management for NuGet, npm, Maven).

A YAML pipeline is structured hierarchically into: **Pipeline** (the root workflow), **Stages** (major logical boundaries like Build, Test, Deploy), **Jobs** (execution blocks running on a single agent), and **Steps** (individual tasks, scripts, or actions run sequentially inside a job).

Microsoft-Hosted agents are fresh, isolated virtual machines managed entirely by Microsoft that spin down after each job run. Self-Hosted agents are dedicated virtual or physical machines managed by you, providing full control over cached dependencies, tools, and local network access.

Service Connections are secure abstracts used to store credentials and authenticate Azure DevOps agents to external systems like Microsoft Azure Subscriptions, SonarQube, GitHub, or Docker registries without exposing plaintext secrets in code.

Variable Groups store key-value configurations and secret keys that can be shared across multiple pipelines. Secure Files are used to store sensitive binary files, such as SSH private keys or Apple distribution certificates, securely within Azure DevOps without tracking them in git.

A multi-stage pipeline defines both continuous integration (CI) and continuous delivery (CD) inside a single YAML file. It allows engineers to visually track code progression from unit tests and building artifacts all the way through Dev, QA, and Production environments in sequence.

You use the `condition` property accompanied by built-in expressions. For example, setting `condition: succeeded('BuildStage')` or `condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))` ensures a task only executes if preceding blocks succeed on the main branch.

Environments represent a logical collection of target deployment resources (e.g., Kubernetes clusters, VMs). Registering them enables deployment tracking, historical traceability back to specific commits, and the enforcement of explicit approval gates or checks.

A **Rolling** strategy replaces instances of the old application version with the new version step-by-step across a target cluster. A **Canary** strategy deploys the new version to a tiny fraction of servers first, allowing you to test stability against live user traffic before initiating a full rollout.

Approvals and Checks are configured via the Azure DevOps UI directly on Service Connections, Agent Pools, or Environments. They intercept pipeline execution, forcing a pause until authorized users sign off or external automated API checks clear.

A `script` (or `bash`/`pwsh`) block executes inline terminal commands directly on the host operating system. A `task` is a packaged, reusable block of abstracted code built by Microsoft or third parties to execute complex actions easily (e.g., `AzureResourceManagerTemplateDeployment`).

Pipeline caching allows you to store files that take a long time to download or compute—such as `node_modules` or NuGet packages—across multiple pipeline runs. If a cache key (typically a hash of a package lock file) matches, the agent restores the directory instantly, cutting down build times.

A deployment job is a specialized type of pipeline job tailored specifically for pushing code to environments. It inherently provides lifecycle hooks (`preDeploy`, `deploy`, `routeTraffic`, `postRouteTraffic`, `on: failure`) and native support for rolling or canary deployment strategies.

First, emit the variable inside a script step using the logging command syntax: `echo "##vso[task.setvariable variable=myVar;isoutput=true]myValue"`. Then, inside the downstream job, configure a dependency on the first job and fetch it using: `$[dependencies.JobA.outputs['StepName.myVar']]`.

PR triggers automatically execute a validation build pipeline whenever a pull request is opened or updated targeting a protected branch. This ensures that incoming code compiles successfully and passes tests before a developer can merge it.

Templates allow you to define reusable chunks of stages, jobs, or steps in a standalone YAML file. Other pipelines can then reference this file, injecting specific parameters. This avoids duplication (Don't Repeat Yourself) and enforces compliance standards across multiple teams.

You install the SonarQube extension from the Marketplace, configure a Service Connection to your SonarQube server, and add three sequential tasks to your YAML build job: `SonarQubePrepare` (injects configuration), your native code compilation step, and `SonarQubeAnalyze` (analyzes and uploads results).

Branch policies protect key branches (like `main`) by enforcing code quality guardrails. They require a minimum number of successful peer reviews, require a green build pipeline, check for unresolved comments, and prevent unauthorized direct commits.

The `PublishPipelineArtifact` task uploads compiled binaries, packages, or files generated during the build stage up to cloud storage. The `DownloadPipelineArtifact` task retrieves these files in downstream deployment stages, ensuring the exact same binary is deployed across environments.

You use the `AzureKeyVault` task inside your pipeline, or link an Azure Key Vault directly to a Variable Group. This queries Key Vault dynamically at runtime, downloading secrets into localized agent memory as masked environment variables that are scrubbed from all execution logs.

HashiCorp Terraform Interview Prep (41-60)

Infrastructure as Code, state tracking, modules, and lifecycle management.

Terraform is an open-source declarative Infrastructure as Code (IaC) tool. It tracks real-world infrastructure by mapping resource declarations in code to actual deployed API resources using a JSON metadata registry file known as the **Terraform State File (`terraform.tfstate`)**.

Storing state locally causes single-point-of-failure risks, lacks team collaboration capabilities, and can expose plaintext secrets. This is solved by using a **Remote Backend** (e.g., Azure Blob Storage with DynamoDB/Blob leasing for state locking) to secure and centralize access.

The workflow consists of four core steps: 1. `terraform init` (downloads provider plug-ins and initializes backends), 2. `terraform validate` (verifies syntax correctness), 3. `terraform plan` (generates an execution preview detailing additions, changes, or deletions), and 4. `terraform apply` (executes the changes against the cloud provider API).

A `resource` block declares components that Terraform is responsible for creating, modifying, and destroying. A `data` source block provides read-only queries to fetch information from existing infrastructure managed outside of the current Terraform workspace.

**Implicit dependencies** are automatically discovered when a resource references an attribute of another (e.g., `ami = aws_instance.web.id`). **Explicit dependencies** are manually forced by the engineer using the `depends_on` meta-argument when dependencies aren't visible through direct variable mappings.

Workspaces allow an engineer to maintain separate state files for the exact same configuration directory. They are useful for managing distinct infrastructure environments (like Dev vs QA) using the same code, though separating environments via explicit directory layouts is often preferred for critical production boundaries.

A module is a container for grouping multiple resources together into a reusable configuration block. Inputs are handled via standard configuration variables (`variable`), and resources expose internal properties to parent calling configurations using output blocks (`output`).

`terraform refresh` queries the provider APIs to update the state file with any real-world changes that occurred outside of Terraform (configuration drift). It does not alter your actual infrastructure; it only reconciles the local state file.

State locking is a security mechanism supported by remote backends. It locks the state file during active `apply` or `plan` cycles, preventing concurrent pipeline executions from running at the same time and corrupting the state file metadata.

Running `terraform plan` automatically performs a refresh under the hood. It compares your code, the current state file, and actual live infrastructure. It will then generate an execution preview to add or modify resources back to the configuration declared in your code.

`count` loops instantiate resources based on an integer index, creating resources as an ordered array. `for_each` loops accept maps or sets of strings, instantiating resources as a distinct map. `for_each` is preferred for dynamic workloads because deleting an element from the middle of a collection won't trigger re-indexing or re-creation of adjacent resources.

The `terraform.tfvars` file is used to assign actual values to variables declared in your code, separating code logic from environment-specific values. Terraform automatically loads any file named exactly `terraform.tfvars` or matching `*.auto.tfvars` in the working directory.

You can untrack the resource by running `terraform state rm `. This removes the resource from the state file tracking system completely, allowing it to remain intact in your cloud environment while your code configuration can be safely cleaned up.

`terraform import` brings pre-existing, manually created cloud resources under Terraform management. It pulls the infrastructure metadata into your state file. You must also write matching resource code blocks to align with the imported resource properties.

The `lifecycle` block configures how Terraform treats resource operations. `create_before_destroy` alters zero-downtime updates by spinning up a replacement instance before dropping the old one. `prevent_destroy` acts as a guardrail, rejecting any execution plan that would result in destroying a critical asset.

Input variables are parameters injected by users or external automation files to customize executions. Local values (`locals`) are internal, self-contained expressions within the module that allow you to assign names to repetitive code logic or calculations, acting like local constants.

Set the `sensitive = true` argument inside your variable declaration block. Terraform will mask these inputs as `(sensitive)` within console outputs and plans, though they will still be stored in plaintext inside the underlying `.tfstate` file metadata.

Terraform Cloud is a managed application platform that provides centralized remote state management, integrated VCS connections, remote run environments, policy guardrails (Sentinel/OPA), and secure variable management for team collaboration.

`terraform taint` manually marks a resource as degraded, forcing it to be destroyed and recreated on the next plan and apply cycle. It has been deprecated in favor of the more flexible flag option: `terraform apply -replace=""`.

Providers are plugins that translate Terraform declarations into upstream cloud or platform API calls (e.g., AWS, Azure, Google Cloud, Kubernetes). They are downloaded dynamically from the public HashiCorp Registry during initialization.

Git & GitHub Interview Prep (61-80)

Version control, branching workflows, merging mechanisms, and collaboration.

Git is a distributed version control system where every contributor clones a complete local copy of the repository history. SVN is a centralized version control system reliant on a single central server to host and track revision histories.

Git manages tracking across three main areas: 1. **Working Directory** (untracked or modified files currently being edited), 2. **Staging Area / Index** (a snapshot layout of files flagged to be captured in the next commit), and 3. **Commit History / Local Repository** (permanently recorded cryptographic snapshots).

`git fetch` downloads remote metadata and changes into your local repository tracking branches without altering your working code directory. `git pull` performs a `git fetch` followed immediately by a `git merge` to integrate those changes into your active branch.

`git merge` combines two histories together by creating a new "merge commit," preserving historical timeline context exactly as it happened. `git rebase` re-writes commit histories by moving your feature branch commits to sit on top of the target branch's latest commit, creating a clean, linear history.

A merge conflict occurs when distinct changes are made to the exact same line of a file on two different branches, preventing Git from automatically merging them. To resolve it, open the affected file, manually pick between the marked changes, stage the file, and complete the commit.

`git stash` temporarily shelves uncommitted changes in your working directory, reverting it to match the clean `HEAD` state. This is useful when you need to quickly switch branches to work on an urgent fix without forcing an incomplete commit.

`--soft` moves the branch pointer back but leaves your staging area and working directory intact. `--mixed` (default) rolls back commits and clears the staging area, but preserves your actual code changes. `--hard` resets everything, destroying all uncommitted work and modified states back to the target commit reference.

`git cherry-pick ` selects a specific commit from an entirely separate branch and applies it as a brand-new commit onto your currently active branch, allowing you to pull separate isolated changes without merging the entire history.

The `.gitignore` file contains wildcard patterns and explicit paths of files and directories that Git should completely ignore from tracking, keeping compile artifacts, dependencies (e.g., `node_modules`), or localized configuration files out of repository histories.

Git Flow is a strict branching framework that organizes project delivery across specific tracking roles: `main` tracks production code, `develop` tracks integrated pre-production builds, `feature/` branches handle new capabilities, `release/` branches prepare production updates, and `hotfix/` branches quickly patch production issues.

Trunk-Based Development is a modern practice where all developers merge small, frequent commits into a single central branch (the "trunk"). It prioritizes CI/CD automated validation and short-lived branches to avoid long merge conflicts, contrasting with the long-lived branches used in Git Flow.

A Pull Request is a feature hosted on collaboration suites like GitHub. It serves as a formal review request, allowing team members to review your proposed changes, discuss modifications, run automated tests, and provide code quality sign-offs before merging the code.

`git reset` alters history by moving branch pointers backwards, which can disrupt shared public branches. `git revert` safely record a brand-new commit that implements the exact inverse changes of a targeted past commit, reversing its effects without changing repository history.

Both are modern YAML-driven automation platforms. Azure Pipelines is optimized for enterprise management, featuring advanced built-in deployment gates and environment tracking. GitHub Actions is deeply integrated into the GitHub ecosystem, relying on event hooks triggered directly within the repository platform.

`git log` prints the repository commit history. It can be streamlined using flags like `git log --oneline --graph --decorate`, or filtered via authors and dates using options like `git log --author="Name" --since="1 week ago"`.

Git hooks are custom execution scripts located in `.git/hooks/` that run automatically when specific lifecycle events occur. For example, a `pre-commit` hook can run lint checks or secret scanning locally before Git allows a developer to complete a commit.

A detached HEAD state occurs when you checkout a specific commit hash or tag directly rather than an active tracking branch name. To fix it, you can run `git checkout ` to return to your work, or create a new branch from that commit to save any experimental changes.

A squash merge condenses all individual commits from a feature branch into a single commit before applying it to the main branch. This combines a series of intermediate commits into one clean entry on the main branch timeline.

Git submodules allow you to keep a separate Git repository pinned as a subdirectory inside a parent repository. This lets you isolate external dependencies or shared libraries within their own standalone version histories.

You use `git bisect`. It performs a binary search through your commit history. You mark a known bad commit and a known good commit, and Git checks out intermediate points for testing until it identifies the exact commit that introduced the bug.

Linux OS & Administration Prep (81-100)

Linux internals, file systems, process states, and networking administration.

The Linux OS architecture comprises four key layers: the **Hardware** layer, the **Kernel** (the core engine managing hardware resources and abstractions), the **Shell** (the command-line interface interpreter), and **User Applications** running on top.

Linux organizes components logically under `/`: `/bin` contains essential binaries, `/etc` stores system configuration scripts, `/var` stores variable data like logs, `/home` holds user directories, and `/proc` is a virtual file system mapping real-time kernel processes.

An Index Node (Inode) is a metadata structure on a storage drive that stores key properties about a file, including its size, owner permissions, and disk block pointers. It does not store the file's name or its actual raw data contents.

A **Hard Link** creates an alternative name pointer pointing directly to the exact same underlying Inode on the disk. A **Soft Link (Symlink)** is a shortcut file containing a path reference pointing to another file name. If the source file is deleted, a symlink breaks, but a hard link remains valid.

Processes exist within several lifecycle states: **Running/Runnable (R)** actively executing code, **Interruptible Sleep (S)** waiting on an event, **Uninterruptible Sleep (D)** waiting directly on hardware I/O operations, **Stopped (T)** halted by signals, and **Zombie (Z)** terminated but still retaining an entry in the process table.

A zombie process has finished executing but its parent process has not yet read its exit status via the `wait()` system call. Zombie processes cannot be killed directly using `kill -9` because they are already dead; you must kill or restart the parent process to clean them up.

Permissions are split across three groups: User, Group, and Others, using numerical values (Read=4, Write=2, Execute=1). `chmod 755` translates to: User gets full access (4+2+1=7), Group gets read and execute (4+1=5), and Others get read and execute (4+1=5).

The file `/etc/fstab` contains configuration details for disk drives, block devices, and remote shares. It defines how and where these devices should be mounted automatically during system boot sequences.

Real-time resource utilization can be monitored using interactive commands like `top` or `htop` for processing overhead, `free -m` for memory allocations, `df -h` for storage tracking, and `iostat` for checking input/output disk metrics.

SysV init processes startup scripts sequentially one by one, resulting in slower boot cycles. `systemd` is a modern initialization daemon that parallelizes service activation, uses target units, and manages process groups using control groups (cgroups).

Network port allocations and socket status can be inspected using commands like `ss -tulnp` or `netstat -tulnp` to identify listening processes, and tools like `nc` (netcat) or `telnet` to test outbound connections to remote network ports.

Load average displays three values indicating CPU and disk I/O queue resource demands over the past 1, 5, and 15 minutes. A load average value of 1.0 on a single-core system indicates 100% capacity utilization.

Swap space is a designated region on a storage drive used when physical RAM is fully saturated. The kernel swaps inactive memory pages out to swap space to free up memory for active workloads, preventing out-of-memory crashes.

`grep` is used to search and extract lines matching a text pattern. `sed` is a stream editor used to parse and transform text. `awk` is a complete programming language designed for advanced data manipulation and column-based text processing.

SSH is secured via `/etc/ssh/sshd_config`. Best practices include changing the default port 22, disabling plaintext password logins to require cryptographic SSH key pairs, and disabling root login capabilities (`PermitRootLogin no`).

A **Soft Limit** is the default resource restriction value enforced for a user or process, which the user can manually increase up to the hard ceiling. A **Hard Limit** represents the maximum restriction ceiling, manageable only by root users.

A cron job is an automated background task scheduled to run at specific times. It is configured using the `crontab -e` command, using five field flags to define scheduling: Minute, Hour, Day of Month, Month, and Day of Week.

Environment variables store global system configuration values used by shells and child processes. To make them permanent for a user, declare the export statement (e.g., `export MY_VAR="value"`) inside their user profile configuration script, such as `~/.bashrc` or `~/.zshrc`.

The OOM Killer is an automated kernel subsystem protective feature. When system memory is completely exhausted, the OOM Killer reviews process scores and terminates low-priority or high-memory consuming processes to protect the core operating system from a total crash.

System logs managed by systemd are reviewed using `journalctl`. You can refine your search using flags like `journalctl -u nginx` to check specific service units, or use time-bounding constraints like `journalctl --since "1 hour ago"`.
